# ADR-037: Sales and Activities capability adapters

- Status: Accepted
- Date: 2026-07-11
- Phase: 6F

## Context

The Sales and Activities owner modules already define validated aggregates and independent persisted-state codecs. Published commands, responses and events are defined by the canonical Protobuf source set and generated by `crm-proto-contracts`. The platform still needs a boundary that maps those wire contracts to owner-domain commands and then to one transaction-aware PostgreSQL plan without adding infrastructure dependencies to either owner module.

Hand-written duplicate wire structs, client-supplied aggregate snapshots, direct SQL in business modules and cross-imports between Sales and Activities would weaken the module and authorization invariants.

## Decision

Add independent platform adapter crates:

- `crm-sales-capability-adapter`;
- `crm-activities-capability-adapter`;
- shared deterministic conversion support in `crm-capability-plan-support`.

Each adapter implements `TransactionalAggregatePlanner`. Target resolution and planning are synchronous and perform no I/O. The PostgreSQL aggregate executor remains the first awaited operation after live authorization, locks the authoritative owner-module record, and supplies its validated snapshot to the planner.

Adapters:

1. decode only generated command messages from `crm-proto-contracts`;
2. verify exact owner, schema version, encoding, data class and descriptor-closure hash;
3. reject cross-tenant resource references before domain planning;
4. convert exact money through canonical base-10 `i128` minor units and never use binary floating point;
5. invoke only public owner-domain constructors and mutations;
6. persist only the independent internal schemas `crm.sales.deal.state@1.0.0` and `crm.activities.task.state@1.0.0`;
7. emit generated versioned Protobuf responses and outbox events;
8. derive deterministic event ids, delivery deduplication keys and canonical audit envelopes from the validated execution context;
9. bind idempotency to the exact capability id, version and semantic input hash.

The generated message descriptor hash is versioned and covers the containing descriptor file plus its complete imported descriptor closure. An unrelated Protobuf package therefore does not change a Sales or Activities payload identity, while changes to referenced shared types do.

## Consequences

- Sales and Activities remain infrastructure-free and do not import one another.
- Public wire compatibility and internal persisted-state compatibility remain independently versioned.
- Corrupt persisted bytes are rejected by the owner codec and mapped to a safe internal error.
- Optimistic versions are evaluated against the PostgreSQL-locked aggregate, not against a caller snapshot.
- Query paths, the public mutation composition, the Sales-to-Activities link and rebuildable projections remain separate reviewable slices.
